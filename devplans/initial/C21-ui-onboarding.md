# C21. UI-Onboarding 개발계획서

| 항목 | 내용 |
|------|------|
| 구성요소 ID | C21 |
| 이름 | UI-Onboarding (최초 실행 마법사) |
| 계층 | UI (PySide6) |
| 관련 Phase | P3(핵심) |
| 상위 문서 | `README.md` |

---

## 1. 목적/책임
첫 실행 시 사용 준비(Provider/모델/권한/핫키)를 단계별로 안내해 진입 장벽을 낮춘다.

- Provider 선택(로컬/클라우드) 및 준비.
- 모델 다운로드 또는 오프라인 경로 지정.
- 마이크 권한/장치 테스트.
- 커서 주입 테스트(한글 확인).
- 핫키 지정.

**책임이 아닌 것**: 실제 다운로드 로직(C18), 키 저장(C19), 설정 저장(C11) — 각 구성요소 호출만.

---

## 2. 선택 근거와 대안 검토
- **온보딩 필요**: 모델 다운로드/키/권한/핫키 등 초기 설정이 많아 미안내 시 이탈 위험.
- **재진입 가능**: 온보딩 없이도 설정(C14)에서 동일 작업 가능해야 함(온보딩은 편의).

---

## 3. 공개 인터페이스 계약
- `start_onboarding()` / `is_completed() -> bool`.
- 각 단계는 기존 구성요소 호출: `download_model`, `test_connection`, `test_mic`, `test_injection`, `set_hotkey`.
- 완료 시 `on_completed` → 설정에 완료 플래그 기록(C11).

---

## 4. 입력/출력 데이터와 상태 전이
- 입력: 사용자 선택.
- 출력: 설정/키/모델 준비 완료 상태.
- 단계 전이: Welcome → Provider → Model → Mic → Injection → Hotkey → Done. 각 단계 건너뛰기 허용(선택).

---

## 5. 의존 구성요소와 호출 관계
- **의존**: C18 ModelManager, C2/C3(연결 테스트), C1(마이크 테스트), C5(주입 테스트), C9(핫키), C11(완료 플래그), C19(키).

---

## 6. 내부 설계
### 6.1 구조
- `onboarding_wizard.py`: 단계 컨트롤러.
- `steps/`: 단계별 페이지(provider/model/mic/injection/hotkey).
- `checks.py`: 각 단계 준비 상태 판정.

### 6.2 단계별 동작
- Provider: 로컬 우선/클라우드 선택 → 키 입력(클라우드 시).
- Model: 권장 모델 다운로드(진행률) 또는 오프라인 경로.
- Mic: 장치 선택 + 레벨 테스트.
- Injection: 테스트 입력창에 샘플 한글 주입 확인(핵심 검증 UX).
- Hotkey: 기본 핫키 지정/충돌 확인.

### 6.3 원칙
- 각 단계는 실패해도 진행 가능(나중에 설정에서 완료), 단 권장 상태 안내.
- 네트워크/다운로드는 워커 + 진행 표시.

---

## 7. 오류/폴백/재시도 정책
- 다운로드/연결 실패: 재시도 또는 "나중에 설정" 안내.
- 주입 테스트 실패: 권한/앱 제약 안내(C5 문서 참조).

---

## 8. 성능 목표
- UI 반응성 유지, 다운로드 등 장시간 작업은 백그라운드.

---

## 9. 보안/프라이버시 고려사항
- 키 입력은 C19 저장, 화면 마스킹.
- 클라우드 선택 시 데이터 전송 고지.

---

## 10. 테스트 항목
- 단위: 단계 전이, 건너뛰기, 완료 플래그.
- 통합: 각 단계가 실제 구성요소 호출/상태 반영.
- 엣지: 오프라인, 다운로드 실패, 주입 실패, 중도 종료 후 재진입.

---

## 11. 관련 Phase와 구현 범위
- **P3**: 전체 온보딩 구현(모델/키/권한/핫키/주입 테스트).

---

## 12. 미결정 사항
- 필수 단계 vs 선택 단계 구분.
- 권장 기본 provider/모델(README 14절 결정 종속).
- 온보딩 재실행 진입점.
