# C22. Updater 개발계획서

| 항목 | 내용 |
|------|------|
| 구성요소 ID | C22 |
| 이름 | Updater (업데이트) |
| 계층 | 통합/배포 |
| 관련 Phase | P5(핵심) |
| 상위 문서 | `README.md` |

---

## 1. 목적/책임
새 버전 확인/안내/설치를 담당한다. 초기에는 수동 안내, 이후 단계적으로 자동화.

- 버전 확인(원격 매니페스트).
- 릴리스 노트 표시.
- 업데이트 다운로드/실행(단계적).

**책임이 아닌 것**: 빌드/설치본 생성(C16), 모델 다운로드(C18).

---

## 2. 선택 근거와 대안 검토
- **후순위 독립 구성요소**: MVP엔 불필요, 배포/상용 단계에 필요. 코드사인/무결성 검증 전에는 자동 설치보다 수동 다운로드 안내가 안전.
- 대안(스토어 자동 업데이트/MSIX): 배포 방식 결정에 종속(README 미결정).

---

## 3. 공개 인터페이스 계약
- `check_for_update() -> UpdateInfo?`.
- `get_release_notes(version) -> text`.
- `download_update(on_progress) -> path`(단계적).
- `apply_update(path)`(단계적, 검증 후).

`UpdateInfo` = { latest_version, url, checksum, mandatory?, notes }.

---

## 4. 입력/출력 데이터와 상태 전이
- 입력: 원격 버전 매니페스트.
- 출력: 업데이트 알림/설치본.
- 상태: idle → checking → available/none → downloading → ready → applied.

---

## 5. 의존 구성요소와 호출 관계
- **의존**: 네트워크, C16(설치본 형식/버전), C20(로깅), C11(자동확인 설정).
- **호출자**: C12/C14 UI(수동 확인/알림).

---

## 6. 내부 설계
### 6.1 구조
- `updater.py`: 확인/다운로드/적용 흐름.
- `manifest.py`: 원격 버전 매니페스트 파싱.
- `verify.py`: checksum/서명 검증.

### 6.2 절차(단계적)
- 1단계(P5 초기): 버전 확인 + 릴리스 노트 + "다운로드 페이지 열기"(수동 설치).
- 2단계(신뢰 확보 후): 설치본 다운로드 + 검증 + 실행.
- 3단계(코드사인 후): 자동 업데이트 옵션.

### 6.3 원칙
- 무결성 검증(checksum, 가능 시 서명) 통과 전 실행 금지.
- 강제 업데이트(mandatory)는 신중히(기본 선택적).

---

## 7. 오류/폴백/재시도 정책
- 확인 실패: 조용히 무시 + 다음 기회(방해 금지).
- 다운로드/검증 실패: 수동 안내로 폴백.

---

## 8. 성능 목표
- 백그라운드 확인, UI 방해 없음.

---

## 9. 보안/프라이버시 고려사항
- 서명/체크섬 검증 필수(변조 방지).
- 버전 확인 시 최소 정보만 전송(프라이버시).

---

## 10. 테스트 항목
- 단위: 매니페스트 파싱, 버전 비교, 검증 로직(mock).
- 통합: 확인→알림→(수동)설치 흐름.
- 엣지: 네트워크 없음, 손상 매니페스트, checksum 불일치.

---

## 11. 관련 Phase와 구현 범위
- **P5**: 버전 확인 + 릴리스 노트 + 수동 설치 안내(1단계). 자동화는 코드사인 준비 후.

---

## 12. 미결정 사항
- 매니페스트 호스팅 위치.
- 자동 업데이트 활성 시점(코드사인 종속).
- 강제 업데이트 정책.
